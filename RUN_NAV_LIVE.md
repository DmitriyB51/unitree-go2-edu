## ⭐ ГЛАВНОЕ, ЧТО ВЫЯСНЕНО НА ЖИВОМ РОБОТЕ (2026-07-23)

**Собака ЕЗДИТ по `/cmd_vel`.** Цепочка Nav2 → `/cmd_vel` → `vel_ctrl` → `/api/sport/request` → лапы
работает. Что для этого понадобилось:

1. **`StopMove` при нулевой команде, а НЕ `Move(0,0,0)`.** Это ЕДИНСТВЕННОЕ, что мешало.
   Пока мост слал `Move(0,0,0)` в покое, робот только **наклонял корпус** вместо шага — и дважды
   сложился. Образец — `pathFollower.cpp:431-437` (CMU), исправлено в `vel_ctrl_repub.cpp`.
   При запуске моста собака теперь слегка оседает — это нормально, `StopMove` меняет стойку.
2. **Статический TF `aft_mapped→body`.** Point-LIO штампует облака кадром `body`, которого в TF не
   было ни у кого → локальная costmap молча выбрасывала ВСЕ облака → живых препятствий не было
   вообще. Добавлено в `tf_setup.launch.py`.
3. **Локальная costmap переведена в кадр `map`** (`camera_init` наклонён на 19.9°, потому что
   Point-LIO работает без акселерометра) + `obstacle_min_range: 0.8` (иначе робот видит свои лапы).

**⛔ НЕ ТРОГАТЬ `ContinuousGait` (1019) и `SwitchJoystick` (1027).** Мы их дёргали наугад — собака
упала и потеряла плавность вставания. **Перезагрузка робота возвращает чистое состояние** и это
правильная первая реакция на любое странное поведение.

**⛔ Любую смену режима — только на ЛЕЖАЩЕЙ собаке.** Оба падения произошли при смене состояния у
стоящего робота.

## ⭐⭐ У ЭТОЙ СОБАКИ ЕСТЬ ПОРОГИ ИСПОЛНЕНИЯ — занижать параметры НЕЛЬЗЯ

Три отказа за сессию, все одной формы: **Nav2 исправно шлёт команды, `/cmd_vel` выглядит живым,
а робот не двигается.** Два из трёх — результат занижения «ради безопасности».

| параметр | НЕ работает | работает |
|---|---|---|
| `desired_linear_vel` | 0.15 м/с (идёт плохо) | **0.3** |
| первая команда поворота | 0.075 рад/с | **0.4** |
| `vx` при развороте на месте | 0 (дефолт RPP) | **0.10** (`ROTATE_ASSIST_VX`) |

**Дедлок разворота — самое неочевидное.** RPP ограничивает команду поворота относительно
**измеренной** угловой скорости: первая команда = `max_angular_accel · dt`, dt = 1/20 = 0.05 с.
При `max_angular_accel: 1.5` это **0.075 рад/с** → собака не шевелится → одометрия 0 →
команда навсегда залипает на 0.075. **Правило: `max_angular_accel >= rotate_to_heading_angular_vel / dt`**
(здесь 0.4/0.05 = **8.0**). Это порог, а не «плавность» — не снижать.

**`ROTATE_ASSIST_VX = 0.10`** в `vel_ctrl_repub.cpp`: RPP при развороте жёстко ставит `vx=0`,
а у собаки на чистом вращении **застревают лапы**. Мост добавляет немного «вперёд» только когда
команда — чистое вращение. Цена: ~0.8 м увода за разворот на 180°.

**➜ Если что-то не едет, а команды идут — первым делом смотри ВЕЛИЧИНУ команды.**

## 🔧 Что ещё надо чинить (не сделано осознанно)

- **Походка при навигации выглядит «пьяной»** — до целей доезжает, но идёт неровно. Не
  диагностировано. Кандидаты по порядку: `ROTATE_ASSIST_VX` мешает довороту; RPP качает
  `angular.z` вокруг пути; `lookahead_dist: 0.6` маловат для 0.3 м/с; частота команд 20 Гц
  против частоты походки.
- **Не вписывается в поворот коридора** (fitness при этом был лучше 0.1 — это НЕ локализация).
  Здесь настоящий конфликт, а не просто кривой параметр:
  * `regulated_linear_scaling_min_speed: 0.3` равен `desired_linear_vel: 0.3` → **сброс скорости
    на кривизне фактически отключён**, робот входит в угол на полном ходу. Поставлено так,
    чтобы RPP не нырял в «мёртвую зону» 0.1–0.2 м/с;
  * **и просто понизить нельзя** — ниже ~0.3 м/с собака идёт плохо (см. пороги выше). Обычное
    лечение «притормозить в повороте» недоступно;
  * **остаётся поворачивать круче.** Радиус = `v/ω`: при 0.3 м/с угол радиуса 0.5 м требует
    **0.6 рад/с**, радиуса 0.3 м — **1.0**. `max_angular_vel` не задан (дефолт RPP 1.0), команду
    Nav2 выдаст, но **исполняет ли собака 0.6–1.0 рад/с на ходу — не проверено**, проверяли
    только 0.4 рад/с на месте;
  * третий подозреваемый — `lookahead_dist: 0.6`: чистая погоня срезает углы по построению,
    а 0.6 м против полуширины коридора ~0.85 м это много.
  * **⚠️ Не угадывать — померить.** Один заезд с записью `/plan`, реальной траектории `base_link`
    и `/cmd_vel` в один CSV: пик поперечной ошибки **на углу** = срезание (лечить `lookahead_dist`),
    равномерный рост = собака не отрабатывает угловую скорость.
- **Аппаратного E-STOP нет** — цели без человека рядом не запускать.
- **Карта expF не подключена** — `building_expF.pgm` геометрически лучше, но для неё **нет `.pcd`**,
  а менять 2D отдельно от 3D нельзя (планировщик и матчер окажутся в разных геометриях).

## 3. Ступень B — тесты, которые можно сделать ТОЛЬКО на роботе

### B1. ✅ ПРОВЕРЕНО 2026-07-23 — и ответ ПЛОХОЙ: пульт НЕ является E-STOP

> **РЕЗУЛЬТАТ ЖИВОГО ТЕСТА.** Пульт **НЕ перебивает** `/cmd_vel`. Рулить он не мешает и управление
> не забирает. Он умеет только **поднять и положить** собаку.
>
> **Единственный рабочий стоп — программный:**
> * `Ctrl+C` на паблишере `/cmd_vel` — собака встаёт за 0.5 с (таймаут **проверен**: в логе
>   `[NAV2]` → `[STOP]` через ~0.6 с);
> * `Ctrl+C` на `run_vel_ctrl.sh`.
>
> Обрыв WiFi/SSH → `/cmd_vel` протухает → **робот останавливается**, а не убегает. Это единственная
> страховка, и она работает.
>
> **Положить собаку пультом** во время движения — ближайший аналог аппаратного стопа: этот канал
> лежит НИЖЕ уровня джойстика (продолжал работать даже после `SwitchJoystick(false)`).
>
> ⚠️ **Мануал Unitree:** одновременная отправка высокоуровневых API-команд и команд с пульта ведёт к
> нестабильности. Это не решено. **Автономные цели без человека рядом не запускать.**

### B1-историческое (как тестировали)

Заводской пульт командует бортовым sport-контроллером
**напрямую (прошивка)**, в обход нашего моста — наш программный арбитраж его не видит.

```bash
~/run_vel_ctrl.sh           # мост запущен, но Nav2 цели не даём
# в другом окне — крошечная ручная команда:
ros2 topic pub -r 10 /cmd_vel geometry_msgs/msg/Twist '{linear: {x: 0.1}}'
```
Робот пошёл медленно вперёд. Теперь:
- [ ] дёрнуть стик пульта → **робот слушается пульта?**
- [ ] нажать стоп/демпфирование на пульте → **робот встал?**

**Если пульт перебивает — отлично, это твой аппаратный E-STOP.**
**Если НЕТ — опасно:** останов только `Ctrl+C` моста; тогда дальше не идти, пока не найдёшь
надёжный стоп.

### B2. Проверить источник препятствий costmap

Мы поменяли его с мирового `/registered_scan` на сенсорный `/cloud_registered_body`
(иначе costmap считает сенсор в (0,0), не чистит лучами и лепит фантомные препятствия →
RPP прерывается с «collision ahead»). Офлайн проверить было нельзя — топика нет в заезде.

```bash
ros2 topic hz /cloud_registered_body                      # он вообще публикуется?
ros2 topic echo /cloud_registered_body --field header.frame_id --once   # какой кадр?
```
- [ ] топик есть
- [ ] `frame_id` — кадр, который **движется** с роботом (тело/`aft_mapped`), не `camera_init`
- [ ] в RViz local costmap не заполнен «стенами» там, где пусто; если пол читается как
      препятствие — поднять `min_obstacle_height` (сейчас −0.30 в `nav2_params.yaml`)
- [ ] если топика нет → включить его в Point-LIO, либо вернуть `/registered_scan` и
      **отключить** проверку столкновений: `ros2 param set /controller_server FollowPath.use_collision_detection false`
